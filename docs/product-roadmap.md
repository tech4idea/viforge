# 产品演进路线图

本文记录当前阶段已经达成共识的 viforge 产品演进方向。后续讨论和实现应以这里的路线图为产品判断基准，再结合具体 feature 文档落地。

## 总体结论

viforge 接下来不应继续围绕单点功能堆叠，而应围绕“本地优先、可自定义、Agent 可迭代优化的 AI 协作工作台”演进。首个开源交付版本暂定产品名为 **ViForge**，具体开源发布检查清单见 [ViForge 开源交付检查清单](./viforge-open-source-release-checklist.md)。

ViForge 的定位不是一个只覆盖小说或情景剧的创作工具，也不是另一个通用代码助手桌面壳。它应强调人在环路的创造性协作：用户提供创意、判断、方向和方法论，Agent 负责检索、整理、生成、审稿、返工和沉淀；系统持续积累可复用的 Agent、技能、知识库、评测案例和个人工作方法。

与 Cloud Code Desktop、Work Buddy、小龙虾、爱马仕等更成熟的通用工作台或代码协作产品相比，ViForge 的差异化不应放在“也能聊天、也能操作文件、也能调用浏览器”上，而应放在：

- 面向创意生产和知识工作，而不只面向 coding 或办公自动化。
- 强调人在环路，用户持续提供判断、品味、创意方向和质量标准。
- Agent、skills、prompt、记忆策略和知识库可以被个人持续积累和迭代。
- 通过 Harness 把 Agent 优化过程变成可复现、可比较、可发布、可回滚的工程闭环。
- 本地优先，用户的工作区、Agent 配置、记忆和方法论资产优先沉淀在自己机器上。

核心方向分为四条线：

1. **桌面单机版产品化**：让用户通过安装包直接启动完整创作环境，不需要理解 Node、API、PostgreSQL、向量库或浏览器代理。
2. **多产品 Profile 化**：继续保持一套 monorepo 和一套基础能力，通过 product profile 区分小说改编、情景剧、学习等不同创作域。
3. **Agent 创作质量闭环**：从“能聊天、能写文件”升级为“能按阶段创作、审稿、返工、保存正式产物”。
4. **Agent Harness 评测闭环**：把 prompt、skill、记忆和工具策略的优化，从主观调参变成可复现、可比较、可发布和可回滚的工程流程。

这四条线的关系是：桌面版解决交付和使用门槛，product profile 解决产品扩展方式，Agent 创作闭环解决用户价值，Harness 解决后续长期优化的质量控制。

## 近期优先级

### P0：开源桌面版可交付

目标是先让 ViForge 成为一个用户可以稳定安装、启动、配置模型并开始协作创作的开源桌面应用。P0 的交付标准不仅是技术链路跑通，还要包含名称、定位、合规性和开源发布体验。

必须完成：

- Windows 安装包稳定启动 Electron、API、静态 Web、内置 PostgreSQL 和 Playwriter relay。
- 数据目录选择、运行配置、API Key 保存和重启生效链路稳定。
- 内置 PostgreSQL binary、pgvector、运行时动态库和打包脚本形成可重复 release 流程。
- 桌面模式的访问 token、单实例锁、关闭/完全退出语义保持清楚。
- 首次启动失败、模型配置缺失、数据库启动失败、Playwriter 未授权等关键失败状态要有可理解的用户提示。
- 产品命名、README、安装包文案和首屏定位统一使用 ViForge，避免继续以内部工程名 viforge 面向用户。
- 开源发布前完成基础合规性检查：项目 license、依赖 license、第三方二进制分发权限、模型/API Key 免责声明、Playwriter 浏览器自动化边界、用户数据本地存储说明。

判断标准：非开发用户安装后，不需要命令行即可完成模型配置并跑通一次完整协作创作会话；开发者看到开源仓库后，能理解 ViForge 的定位、边界和本地优先数据策略。

### P1：创作流程产品化

目标是把当前 LangGraph 主 agent + specialist 委派能力，收敛成用户能感知的创作流程。

必须完成：

- 对正式创作请求展示清楚的阶段：脑暴、故事、审稿、返工、剧本、最终保存。
- reviewer 输出结构化结论：`passed / rejected / targetAgentId / reasons / revisionRequirements`。
- 主 agent 根据 reviewer 结果自动返工，并遵守最大返工轮次。
- 未通过质量闸门的故事或剧本不写入正式产物路径。
- 前端展示当前阶段、调用的 specialist、返工轮次和失败原因，而不是只展示底层工具事件。

判断标准：用户提出正式创作请求后，系统能完成从创意到正式文件的闭环，且用户能看懂每一步为什么继续、打回或停止。

### P2：Harness 驱动 Agent 迭代

目标是让 Agent 优化可以被记录、复现、比较和发布，避免 prompt 修改只靠体感。

必须完成：

- 从失败 run 快速生成 EvalFixture 的工作流更顺畅。
- PromptBlock diff、AgentSpec candidate/active 对比和发布状态机继续打磨。
- EvalRun 的断言失败点能定位到文件 diff 和人工备注。
- release record 支持更清楚的筛选、回滚和审计说明。
- 敏感工具输出、记忆、知识检索的 fixture 裁剪策略可配置。

判断标准：每次重要 Agent 行为变更，都能用 fixture 和人工评分说明“为什么可以发布”。

### P3：多产品扩展

目标是在不复制应用的前提下，稳定维护小说改编、情景剧和后续产品形态。

必须完成：

- product profile 继续作为领域差异的唯一主入口。
- 工作区模板、system agent、specialist prompt、manifest、artifact path 和默认知识库都从 profile 派生。
- 普通 run 和 EvalRun 都记录 resolved product config，保证历史结果可解释。
- 新产品只新增 profile、prompt、模板和 manifest，尽量不改基础 Web/API 代码。

判断标准：新增或调整一个产品形态时，不需要 fork `apps/web` 或 `apps/api`。

## 暂不优先

以下方向有价值，但不应抢在 P0/P1 前面：

- 复杂多人协作、账号体系和云端权限。
- 完整 LLM judge 自动评分体系。
- 大规模素材资产管理和媒体生产流水线。
- 过早抽象成通用 Agent 平台或插件市场。
- 维护多个长期分支或复制多套应用。

当前更重要的是先把本地桌面交付、创作闭环和可验证迭代跑稳。

## 后续讨论焦点

下一轮产品讨论建议围绕以下问题展开：

1. 首个正式交付版本主打小说改编、情景剧，还是继续保留多 profile 但只突出一个默认入口。
2. 桌面版首次启动流程要做到多强：只配置模型，还是也引导创建第一个项目和导入素材。
3. 创作闭环里哪些阶段必须自动完成，哪些阶段应该让用户确认后再进入下一步。
4. Harness 在真实产品里是开发者后台、隐藏实验室入口，还是未来面向高级用户开放。
5. 微信入口的定位是轻量创作入口、通知入口，还是远程控制桌面工作台的入口。
