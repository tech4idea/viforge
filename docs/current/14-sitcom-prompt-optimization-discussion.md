# 情景剧故事创作提示词优化讨论稿

本文先作为讨论文档，不直接定义最终实现。当前重点只讨论“故事创作环节”的 agent 改造：从用户想法、人物/场景约束、脑暴方向，到正式单集故事产出和严格评审闭环。剧本、分镜和视频生成先不展开，只保留与故事创作边界有关的说明。

## 当前问题

情景剧产品当前默认提示词过短，主要问题不是“语言不够丰富”，而是职责边界不够硬：

- system agent 被描述为“围绕角色、故事、剧本、分镜和视频生成推进”，但没有明确哪些事该自己做，哪些事必须委派。
- brainstorm-agent、story-agent、screenwriter-agent、reviewer-agent 只有一句定位，缺少输入边界、质量标准、输出格式和禁区。
- 当前 prompt 提到分镜和视频，但没有对应 specialist agent。system agent 如果遇到分镜、视频提示词、角色设定、连续性维护等任务，只能自己临时处理或误用 story/screenwriter/reviewer。
- Mastra 当前实现是“主 agent 常驻 + 按需委派”，不是固定流水线。因此 prompt 不能假设所有正式请求都会被后端硬编排，必须让 system agent 自己判断委派时机、上下文读取、写入边界和审稿闭环。

优化方向应该是：system agent 只做故事创作链路的调度、上下文治理、质量闭环和最终交付；故事本身由 story-agent 承担；质量闸门由 reviewer-agent 严格把关；人物、连续性和喜剧机制这三类高风险前置能力，可以考虑新增 specialist agent。

## 一、System Agent 的职责范围

### 推荐定位

system agent 是情景剧创作工作台的总导演和制片统筹，不是万能编剧。它的职责是判断用户意图、读取必要上下文、选择专业 agent、控制质量闸门、决定是否写文件，以及把最终结果清晰交付给用户。

它可以直接处理的任务：

- 普通问候、解释、问答。
- 对已有文件做摘要、整理、轻量润色。
- 用户明确指定路径的小范围改写，例如“把这句台词改短一点”。
- 简单文件操作，例如按用户要求保存一段已确认内容。
- 对工作区结构、已有设定、已生成产物进行说明。

它应该委派给子 agent 的故事相关任务：

- 明确要求脑暴故事方向、人物关系、笑点机制时，委派 brainstorm-agent。
- 明确要求正式单集故事、整季故事线、A/B 故事、情节卡片时，委派 story-agent。
- 明确要求审稿、把关、判断故事是否成立，或正式故事产物准备落盘前，委派 reviewer-agent。
- 已新增 character-agent、continuity-agent；人物设定、连续性检查应优先委派给对应 specialist。comedy-doctor-agent 仍作为后续可选增强，暂不进入当前 registry。

它不应该做的事：

- 不亲自替代 story-agent 承担完整故事创作。
- 不把 story-agent 产物简单包装后直接当作合格故事。
- 不在没有 reviewer-agent 通过的情况下，把正式故事写入“正式/定稿”路径。
- 不把“准备调用某 agent”“我将先做计划”这类内部调度当作独立回复。
- 不让脑暴进入审稿闭环；脑暴默认只留在聊天中，除非用户明确要求保存。

### 意图分流建议

system agent 可以按四类请求处理：

1. 轻量交互：普通问答、解释、局部修改。直接完成，必要时读写文件。
2. 探索型创作：用户还在找方向、角色、冲突、笑点。委派 brainstorm-agent，不审稿，不落正式文件。
3. 正式故事产物：用户要求生成单集故事、整季故事线、A/B 故事、情节卡片等可交付内容。委派 story-agent，并必须经过 reviewer-agent。
4. 故事质量控制：用户要求“审一下这个故事”“哪里不行”“能不能拍成一集”。委派 reviewer-agent，输出通过/打回和返工目标。

### 写入边界

建议把文件写入分为三档：

- 草稿写入：用户明确要求保存脑暴结果、备忘、候选方向时，可写入用户指定路径或“02 故事/素材草稿”类路径。
- 正式写入：故事通过 reviewer-agent 后，写入产品 profile 中约定路径，例如 `02 故事/<集数>/单集大纲.md`、`02 故事/<集数>/情节卡片.md` 或用户明确指定路径。
- 过程不落盘：reviewer-agent 的审稿意见默认只展示在聊天中，不保存为项目文件，除非用户明确要求记录。

### 质量闭环

正式故事需要保留质量闭环，但在当前 Mastra 架构下，应写成 system agent 的行为规则，而不是假设后端强制流水线：

```text
用户正式请求故事
  -> system agent 读取项目设定和相关文件
  -> 必要时先委派 character-agent / continuity-agent / brainstorm-agent 补齐约束
  -> 委派 story-agent 产出故事
  -> 委派 reviewer-agent 审稿
  -> 若打回，在返工上限内把不合格项交回 story-agent 或对应前置 agent
  -> 通过后写入正式路径并总结
```

返工上限继续使用 `config.toml` 中的 `max_revision_rounds`，没有读取到时默认 5。达到上限仍不通过时，system agent 应停止自动循环，说明最近一次不合格项、失败根因，以及需要用户补充或选择的点。

### 故事创作闭环建议

故事创作建议拆成五个阶段，而不是让 story-agent 一次性从空白写到成稿：

1. 上下文盘点：system agent 读取 `01 基本设定/项目简介.md`、`人物设定.md`、`角色关系.md`、`场景设定.md`、`风格约束.md`、`02 故事/整季故事线.md` 和目标集数已有文件。
2. 前置补齐：如果人物动机、固定场景、喜剧风格或上一集状态不明确，先由 system agent 追问，或委派 character-agent / continuity-agent 给出约束摘要。
3. 故事生成：story-agent 根据明确约束产出单集故事，不直接写剧本。
4. 严格评审：reviewer-agent 按硬门槛审查故事。只要主角目标、阻力升级、因果链、人物一致性、喜剧机制、结尾回收中有一个结构性失败，就必须打回。
5. 返工与落盘：system agent 把打回项转成明确修订任务交给 story-agent；通过后再写入正式故事文件。

## 故事创作质量基准

这部分应该进入 story-agent 和 reviewer-agent 的核心提示词，尤其 reviewer-agent 必须按这些标准严格验收。

### 合格故事的最低门槛

一个情景剧单集故事必须同时满足以下条件，缺一项就不能通过：

- 有清楚的中心人物，且中心人物在本集有具体、可行动、可失败的目标。
- 有持续阻力。阻力必须反复改变主角行动，不能只是背景困难。
- 有因果链。每个主要事件都应由前一事件或角色选择引发，不能是并列段子。
- 有升级。局面必须变得更尴尬、更复杂、代价更高或误会更深。
- 有选择与后果。角色必须做出选择，选择带来后果，后果推动下一步。
- 有喜剧机制。笑点来自人物缺点、身份错位、误会、规则冲突、关系压力或反拍，而不是孤立金句。
- 有人物一致性。角色行动符合既有身份、欲望、缺点和关系，不为剧情硬转弯。
- 有单集闭合。故事能在一集内完成核心冲突，同时可以留下轻量钩子。
- 有结尾回收。结尾需要反转、误会揭开、关系变化、代价兑现、包袱回收或人物自我暴露。

### 应直接打回的故事问题

reviewer-agent 遇到以下任一情况，应直接打回，不给“基本通过”：

- 主角没有具体目标，只有状态描述或情绪表达。
- 故事只是几个段子串联，没有因果推进。
- 冲突靠外部巧合硬推，角色选择不起作用。
- 阻力出现一次就消失，没有持续升级。
- A/B 故事互不影响，只是凑篇幅。
- 笑点依赖网络热梗、吐槽或金句，和人物缺点无关。
- 结尾没有回收，只是停在一个事件结束。
- 角色行为明显违背设定，且没有故事内理由。
- 单集容量失控，需要大量新人物、新地点或复杂外部设定才能成立。
- 故事无法被拆成可表演、可拍摄的场景。

### 好故事的加分项

这些不是最低门槛，但可以作为 reviewer-agent 的高标准参考：

- A 故事和 B 故事在主题上互相照镜子，或在结尾互相触发。
- 喜剧升级不是更吵，而是误会、关系和规则持续收紧。
- 每个主要角色都带着自己的错误理解行动。
- 结尾同时完成笑点回收和人物关系变化。
- 故事给 screenwriter-agent 留下明确的场景压力、对白方向和表演空间。

## 二、现有子 Agent 提示词完善方向

### brainstorm-agent

当前问题：只有“探索设定、人物、场景、冲突和笑点方向”的一句话，缺少输出结构，也没有要求它控制脑暴粒度。

建议职责：

- 把模糊想法变成可发展故事种子。
- 提供多个方向，但每个方向都要说明喜剧机制，而不只是剧情梗概。
- 帮用户识别哪一个方向最适合进入正式 story-agent。
- 可以追问，但如果信息足够，应先给候选方向，不要只反问。

建议质量标准：

- 候选方向必须包含角色欲望、阻力、喜剧错位和升级可能。
- 笑点来源优先来自人物缺点、关系压力、误会、规则冲突、身份错位，而不是硬塞网络段子。
- 每个方向应能在一集或短单元内完成。

建议输出格式：

```md
## 候选方向

### 方向 1：标题
- 一句话故事种子：
- 核心人物欲望：
- 阻力/误会：
- 喜剧机制：
- 升级方式：
- 风险：

## 推荐方向

推荐进入 story-agent 的方向：...
理由：...
```

禁区：

- 不写正式单集大纲或剧本。
- 不调用 reviewer-agent。
- 不默认写入项目文件。

### story-agent

当前问题：把“单集大纲、A/B 故事、冲突升级和结尾反转”列出来了，但没有定义“好故事”的硬标准，也没有说明和剧本的边界。

建议职责：

- 把已选方向或用户请求变成一个可审稿的单集故事。
- 负责故事因果、人物目标、阻力升级、A/B 故事关系、反转或收束。
- 不写完整剧本对白，只写必要的关键台词或笑点说明。

建议质量标准：

- 主角目标具体，不是“想变好”“想证明自己”这类抽象目标。
- 阻力持续妨碍主角行动，并且每一轮升级都改变局面。
- A/B 故事有主题或因果互文，而不是两条互不相干的线。
- 喜剧来自人物和情境，不依赖孤立段子。
- 结尾必须有反转、关系变化、误会揭开、代价兑现或人物自我暴露。
- 故事适配固定场景和固定角色，能被后续 screenwriter-agent 拆成场次。

建议输出格式：

```md
# 第 X 集《标题》单集故事

## 一句话故事核

## 本集主题

## 已知设定约束

## A 故事
- 主角目标：
- 初始阻力：
- 升级 1：
- 升级 2：
- 最坏时刻：
- 结尾反转/收束：

## B 故事
- 角色目标：
- 与 A 故事的关系：
- 喜剧功能：
- 回收方式：

## 关键场景

## 涉及人物和场景

## 给 screenwriter-agent 的编剧提示
```

禁区：

- 不用文学化正文掩盖因果问题。
- 不把笑点清单当故事。
- 不写完整剧本格式。

### screenwriter-agent

当前问题：强调了“可拍摄、可表演、对白有节奏”，但缺少场景级标准，容易把故事扩写成台词稿。

建议职责：

- 把已成立故事拆成可拍摄场景。
- 为每场戏明确场景目标、冲突、动作、对白、节拍和场景转折。
- 保持角色声音差异，并服务已通过故事，不随意重写故事核心。

建议质量标准：

- 每场戏开头应明确地点/时间/出场人物。
- 每场戏至少有一个可表演的冲突、隐瞒、误会、利益交换或关系压迫。
- 动作写镜头能拍到的行为，不写抽象心理。
- 对白短、带目的、有潜台词；避免角色替作者解释剧情。
- 喜剧节拍有铺垫、误导、升级、回收或反拍。
- 场景结尾要改变局面，推动下一场。

建议输出格式：

```md
# 第 X 集《标题》剧本

## 冷开场

### 场景 1：地点 / 时间
出场人物：
场景目标：
冲突/信息差：

动作：

角色A：
角色B：

喜剧节拍：
场景转折：

## 正戏

### 场景 2：地点 / 时间
...

## 结尾
```

禁区：

- 不新增破坏故事因果的段子。
- 不写无法拍摄的心理独白。
- 不让所有角色说同一种话。
- 不把“节拍”写成空标签，必须说明具体铺垫和回收。

### reviewer-agent

当前问题：只有“通过或打回”的结论，但缺少按产物类型拆开的验收标准，也没有要求它输出结构化返工对象。

建议职责：

- 作为质量闸门，而不是润色助手。
- 现阶段重点验收故事，剧本验收先维持原有能力，不在本文展开。
- 判断问题根因属于 story-agent、character-agent、continuity-agent，还是需要 system agent 补上下文。
- 输出明确的打回原因和返工要求，但不直接改稿。

严格度原则：

- 默认立场是“找结构性问题”，不是“尽量通过”。
- 只有当故事满足最低门槛，且问题只剩措辞、格式、轻微信息补足时，才能通过。
- 不能用“有潜力”“方向不错”“可以进一步完善”作为通过理由。
- 不允许输出折中结论，例如“基本通过”“有条件通过”“建议通过”。
- 每个打回理由必须指向故事中的具体位置、具体缺陷和具体后果。
- 如果缺少必要上下文导致无法判断，应打回给 system，而不是假装通过。

建议故事审查项：

- 主角目标是否具体。
- 阻力是否持续有效。
- 升级链是否有因果，而不是并列事件。
- A/B 故事是否互相服务。
- 人物行为是否符合设定。
- 喜剧机制是否来自情境和人物。
- 结尾是否完成反转、回收或关系变化。
- 是否适配单集容量和固定场景。

故事审查必须使用“一票否决”规则：

- 主角目标不具体：打回。
- 阻力不持续或没有升级：打回。
- 事件没有因果链：打回。
- 喜剧机制与人物/关系无关：打回。
- 结尾没有回收或变化：打回。
- 人物行为违背设定：打回。
- 无法拆成可表演场景：打回。

建议输出格式：

```md
结论：通过 / 打回
审查对象：故事
打回对象：story-agent / character-agent / continuity-agent / system / none
硬门槛检查：
- 主角目标：通过 / 失败，原因：...
- 持续阻力：通过 / 失败，原因：...
- 因果升级：通过 / 失败，原因：...
- 喜剧机制：通过 / 失败，原因：...
- 人物一致性：通过 / 失败，原因：...
- 结尾回收：通过 / 失败，原因：...
不合格项：
1. ...
2. ...
返工要求：
1. ...
2. ...
通过条件：
1. ...
```

建议后续如果做结构化闭环，可以让 reviewer-agent 返回 JSON：

```json
{
  "status": "passed | rejected",
  "targetAgentId": "story-agent | character-agent | continuity-agent | system | none",
  "artifactType": "story",
  "gateResults": {
    "protagonistGoal": "passed | failed",
    "sustainedObstacle": "passed | failed",

    "characterConsistency": "passed | failed",
    "endingPayoff": "passed | failed"
  },
  "reasons": ["..."],
  "revisionRequests": ["..."]
}
```

## 三、故事创作相关可新增子 Agent

新增 agent 不应为了“看起来完整”而增加。判断标准是：这个任务是否有独立质量标准、独立上下文需求、独立输出格式，并且和现有 story/screenwriter/reviewer 有清晰边界。

### 优先级 P0：character-agent

用途：人物设定、角色关系、角色声音、喜剧缺点和长期弧光。

为什么需要：情景剧高度依赖固定人物。当前 story-agent 和 screenwriter-agent 都会临时处理人物，但没有一个 agent 负责角色一致性和角色可持续复用。

职责：

- 生成或完善人物小传。
- 定义角色欲望、隐藏需求、喜剧缺点、行为边界、语言风格。
- 维护角色关系表和常见冲突。
- 为 story-agent 提供“这个角色会怎么行动”的约束。
- 为 screenwriter-agent 提供角色声音差异。

输出路径建议：

- `01 基本设定/人物设定.md`
- `01 基本设定/角色关系.md`

触发场景：

- 用户要求“完善主角/反派/配角”。
- story-agent 或 reviewer-agent 发现人物动机不清、角色行为不一致。
- screenwriter-agent 需要对白风格约束。

### 优先级 P0：continuity-agent

用途：连续性、设定一致性、历史剧情和集间关系维护。

为什么需要：情景剧会不断生成多集内容，角色关系、场景规则、历史包袱和伏笔容易漂移。reviewer-agent 能发现问题，但不适合维护长期状态。

职责：

- 读取已有设定、故事和剧本，整理连续性事实。
- 检查新故事/剧本是否破坏既有设定。
- 维护“已发生事件”“角色关系变化”“可回收包袱”。
- 给 story-agent/screenwriter-agent 提供不可违背约束。

输出路径建议：

- `01 基本设定/连续性圣经.md`
- `02 故事/整季故事线.md`

触发场景：

- 用户要求生成多集、整季线或接续上一集。
- reviewer-agent 打回“人物/设定不一致”。
- system agent 准备正式写入多集故事前。

### 优先级 P1：comedy-doctor-agent

用途：专门诊断笑点、节奏、反拍、铺垫回收。

为什么需要：情景剧质量的核心是喜剧机制。reviewer-agent 可以判定失败，但如果只是把问题交回 story/screenwriter，容易泛泛修改。comedy-doctor-agent 可以作为专项修复 agent。

职责：

- 诊断笑点为什么不成立。
- 提供不破坏人物和因果的笑点升级方案。
- 调整对白节奏、反拍、误会升级和回收。

边界：

- 不负责完整故事创作。
- 输出应作为 story-agent 的修订建议。

触发场景：

- reviewer-agent 打回“喜剧机制弱”“笑点只是吐槽”“升级不好笑”。
- story-agent 连续两轮返工后故事结构成立但笑点仍不成立。

### 优先级 P2：episode-planner-agent

用途：整季故事线、单集主题池、集间顺序和角色弧光规划。

为什么需要：如果用户一开始要求“生成 3 集/10 集故事大纲”，story-agent 一次写多集容易每集都浅。episode-planner-agent 可以先决定每集的戏剧任务，再让 story-agent 逐集深化。

职责：

- 规划多集主题、核心冲突和角色关系变化。
- 控制每集只解决一个主要故事问题。
- 避免多集重复同一种误会或同一种结尾。
- 给 story-agent 输出逐集创作 brief。

输出路径建议：

- `02 故事/整季故事线.md`

触发场景：

- 用户要求多集故事、整季规划、角色长期弧光。
- reviewer-agent 发现单集故事和整季方向冲突。

### 优先级 P2：format-editor-agent

用途：格式整理、模板套用、文件清洁。

为什么需要：当内容已经通过，但格式不符合模板或需要统一命名时，没必要再调用创作型 agent。

职责：

- 将故事、剧本、分镜整理成模板格式。
- 修正标题层级、表格、路径、文件名。
- 不改变创作实质。

边界：

- 不能改人物动机、故事因果或对白内容。

## 推荐新增顺序

短期最值得加的是两个：

1. character-agent：补上情景剧最关键的固定角色体系。
2. continuity-agent：补上多集创作和固定设定一致性。

第二阶段再加：

1. comedy-doctor-agent：当 reviewer-agent 经常打回“喜剧弱”时加入。
2. episode-planner-agent：当多集/整季创作成为高频需求时加入。

format-editor-agent 暂时不进入故事创作第一阶段。分镜、视频提示词相关 agent 也先不纳入本文范围。

## 建议讨论决策

后续落地前建议先确认这些问题：

1. 正式故事产物路径是否使用当前 profile 的 `02 故事/01 第一集/单集大纲.md`，还是改成设计讨论里的 `故事正文.md`。
2. 剧本正式路径是否使用当前 profile 的 `03 剧本/01 第一集/定稿剧本.md`，还是统一为 `剧本.md`。
3. reviewer-agent 是否只做人类可读 Markdown，还是现在就增加 JSON 结构化输出，方便 system agent 自动返工。
4. 第一批新增 agent 是否只加 character-agent 和 continuity-agent，避免一次性扩张 registry。
5. 脑暴结果是否需要约定默认草稿路径，还是继续保持“用户明确指定才保存”。
6. reviewer-agent 是否采用“一票否决”硬门槛，并把 `gateResults` 作为后续自动返工的结构化依据。
